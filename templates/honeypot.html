Honeypot routing (phase 61)

Protek doesn't run a honeypot — operator owns that. What this page configures is the criteria for tagging high-reputation attackers as honeypot-bound. A CF Worker, nginx auth_request, or similar can subscribe to /api/v1/honeypot/targets and route those IPs to the honeypot URL instead of blocking outright. Captured behavior posts back to /api/external/honeypot/callback and feeds the IP's reputation breakdown via tags.

Consumer wiring

The consumer (CF Worker, nginx, etc.) fetches the target list and routes matching requests to the honeypot URL. Bearer token from /admin/tokens with read scope. 

# Fetch the active honeypot target set
curl -H "Authorization: Bearer <token>" {{ request.host_url }}api/v1/honeypot/targets

# Report behavior back to Protek (operator's honeypot makes this call)
curl -H "Authorization: Bearer <token>" \
     -H "Content-Type: application/json" \
     -d '{"ip":"1.2.3.4","action":"credential-probe","detail":"..."}' \
     {{ request.host_url }}api/external/honeypot/callback
