# Troubleshooting

## `/health` returns 503

Body lists the specific issues:

| Issue                | What it means                                                  | Fix |
|----------------------|----------------------------------------------------------------|-----|
| `poll_stale`         | `poller.last_at` is older than the staleness budget            | Check `journalctl -u protek` for a crashed poller thread; or look at `/perf` for a runaway cycle. |
| `lapi_degraded`      | The most recent poll cycle failed                              | `cscli decisions list` to confirm CrowdSec is healthy; check `CROWDSEC_BOUNCER_KEY` matches `cscli bouncers list`. |
| `mt_unreachable`     | MT is configured but the connection probe failed               | `ping`/`telnet` from the VPS to the router on the API port. Check `MT_HOST/MT_USERNAME/MT_PASSWORD`. |
| `poller_disabled`    | `CROWDSEC_BOUNCER_KEY` is unset                                | Generate one (`cscli bouncers add protek`) and paste into `.env`. |
| `poller_not_started` | No completed cycles yet (still booting)                        | Wait ~10s and re-check. |

## Initial sync stuck at the same count

- Look at `journalctl -u protek -n 50` for `reconcile cycle failed:` lines — typically
  an MT API error (auth, network, list-not-found).
- Try `protekctl sync run` to force an immediate cycle and see the response.
- `cscli bouncers list` — Protek's last-pull timestamp should be recent. If it's stuck,
  Protek isn't actually polling.

## Service won't start

```bash
journalctl -u protek -n 100 --no-pager
```

Most common causes:

- Syntax error in `.env` (a value with `=` or quoting weirdness).
- `protek.db` corrupted from a hard crash. Quick repair:
  ```bash
  sqlite3 /var/www/Protek/protek.db "PRAGMA integrity_check;"
  ```
  If it reports anything other than `ok`, restore from your last `/admin/backup` bundle
  (or accept the loss and re-bootstrap).

## "I'm locked out"

You shouldn't be — Protek refuses to lock the env-anchored admin (user #1). If you
truly can't log in:

```bash
sudo bash /var/www/Protek/scripts/setup_admin.py --username syed --rotate-totp-only
```

This regenerates the TOTP secret without touching the password. The new secret is
printed once — capture it.

If the env-anchored admin row got deleted somehow (it shouldn't be possible from
the UI), `seed_env_user()` will recreate it on next boot from `.env`.

## "/admin/backup" import says "decryption failed"

Wrong passphrase, OR the bundle was generated by a different Protek instance using
a different scrypt parameter set. (We pin scrypt n=2^15 r=8 p=1 at v1.0.)

## Webhook DLQ keeps filling up

- `/webhooks` shows the receiver URL + last error. Common: receiver returning 401/403
  (auth issue), 5xx (downstream broken), or connection refused (receiver down).
- After fixing, `Replay` each DLQ entry to ship it.
- Set `event_mask` narrower to reduce noise — `*` includes every auth/sync/decision event.

## SIEM events stop arriving

`/siem` shows the journal + ship state per event. If `shipped_at` is unset on recent
entries, the worker can't reach the syslog/webhook target. The worker auto-reconnects
on TCP socket close; check the receiver side first.

## Reconcile cycle suddenly takes 60s+

Something's slow downstream. Check (in order):

1. `/perf` → "Slowest cycles ever" — what's the duration breakdown? Mostly correlates
   with `to_add` size.
2. If `to_add` is huge: probably a new community blocklist subscription dropped 10k+
   IPs at once. The next ~50 cycles will drain it (batch_cap=200). This is normal.
3. If `to_add` is small but cycles still slow: MT API latency is bad — `tcpdump`/`ping`
   to the router.
4. If neither: SQLite is fighting itself. `du -sh protek.db-wal` — if > 100MB,
   `sqlite3 protek.db "PRAGMA wal_checkpoint(TRUNCATE);"` reclaims it.

## I changed `.env` but nothing changed

- Runtime knobs (sync_interval, batch_cap, dry_run, notify creds, atom_url, etc.)
  live in the settings table and override `.env`. Either edit them at `/settings` /
  `/notifications`, or `set_setting()` them, or delete the override row from `settings`
  table to fall back to `.env`.
- Secrets (`MT_PASSWORD`, `SECRET_KEY`, etc.) only live in `.env`. Editing those needs
  `systemctl restart protek`.

## CrowdSec is detecting attacks but Protek's `/alerts` is empty

The alerts mirror needs **machine credentials**, not just a bouncer key:

```bash
sudo cscli machines add protek-machine --auto
# paste machine_id + password into .env
sudo systemctl restart protek
```

Within ~60s the alerts table should populate (mirror runs every 6 poller cycles).
